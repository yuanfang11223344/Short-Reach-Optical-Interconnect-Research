## 文件命名

日报：

```text
YYYY-MM-DD.md
```

周报：

```text
weekly-YYYY-WXX.md
```

其中 `YYYY-WXX` 使用 ISO week 编号。

## 同步一致性要求

同一个日期或同一个周次的三份 Markdown 报告，正文内容必须一致。如果发布后修正错字、来源链接或技术表述，必须在同一次维护中同步更新三处副本。

## 更新前巡检与补齐

每次更新本专题报告时，必须先巡检近期缺口，再生成当天报告。

巡检步骤：

1. 用 Asia/Shanghai 日期计算今天，检查最近 7 个自然日的日报。
2. 检查最近 7 个自然日覆盖到的所有 ISO 周周报；如果今天是周一，还要检查本周周报。
3. 对每个日报或周报，同时检查 Obsidian、本地阅读副本、专题 Git 仓库三处是否存在。
4. 对三处都存在的文件计算正文一致性；如不一致，优先以专题 Git 仓库文件为 canonical 同步修复。
5. 检查专题 Git 仓库是否存在未提交变更、未推送提交或推送失败遗留。

补齐规则：

- 发现缺失日报或周报时，必须先补齐缺失日期或周次，并同步三处，再生成当天应生成的报告。
- 缺失报告不能用占位文本代替；必须重新检索公开来源并按日报/周报质量标准生成。
- 如果只是一处副本缺失，而另外两处已有一致内容，可以直接复制一致内容补齐，并在执行报告中说明。
- 如果存在本地提交但 GitHub 远端未更新，应先推送；推送失败时保留本地提交并报告错误。

## 周一生成规则

每周一上午 10:00 触发时必须同时产出日报和周报：

1. 先生成当天日报 `YYYY-MM-DD.md`。
2. 再生成本周周报 `weekly-YYYY-WXX.md`。
3. 周报必须读取并纳入当天日报中的 `🔥` 和 `⭐` 条目。
4. 周报不能替代日报；执行报告中必须分别列出日报和周报的保存路径、链接数量和 Git 同步结果。

## 执行报告格式

每次生成或同步后，执行报告需要包含：

- 模式：日报或周报；周一 10:00 触发时必须同时说明日报和周报。
- 检索方向或信源数量。
- 选入事件/条目数量。
- 去重来源链接数量。
- 更新前巡检范围、发现的缺失日期/周次、补齐结果。
- 保存路径。
- 专题 Git 仓库的 commit hash 和推送分支。
